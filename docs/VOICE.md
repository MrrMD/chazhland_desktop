# Голос и демонстрация экрана

Онбординг-документ для разработчика, который входит в проект **chazhland-desktop**.
Описывает голосовую подсистему клиента: вход/выход в голосовой канал, микрофон
(PTT vs голосовая активация, порог гейта), deafen, выбор устройств, эхоподавление/AGC,
демонстрацию экрана (Go Live + просмотр чужой демонстрации, системный звук на Windows),
саундпад и клиентский шумодав RNNoise.

Сначала идёт раздел «Как пользоваться» (для понимания UX), затем — техническое
устройство и настройки.

## 1. Технологический фундамент: LiveKit (self-hosted)

Голос и демонстрация экрана работают поверх **LiveKit** (WebRTC SFU), который мы
хостим сами — без LiveKit Cloud. Клиентский слой реализован в `src/lib/voice.ts`.

- Подключение к комнате (room) выполняется по токену, который выдаёт бэкенд:
  клиент вызывает `api.livekitToken(channelId)`, бэкенд генерирует JWT.
- Идентичность участника в комнате (`identity`) — это `userId` (из JWT-claim).
- Аудио-пресет: `AudioPresets.musicHighQuality` c включёнными **RED**
  (redundancy encoding, устойчивость к потерям) и **DTX** (discontinuous
  transmission, не шлём тишину). Это даёт чистый голос и устойчивость к потере
  пакетов.
- Включены `dynacast` и `adaptiveStream` — оптимизация полосы (SFU не шлёт лишние
  слои/качества тем, кто их не смотрит).

> Имя участника в UI берётся **не** из LiveKit-токена (там `name` часто пустой),
> а из списка участников (member list). При сопоставлении демонстрации экрана с
> человеком матчинг идёт по `userId`.

## 2. Как пользоваться (UX)

### 2.1. Вход и выход из голосового канала

- Голосовые каналы (тип `VOICE`) видны в **ChannelSwitcher** (модалка поиска
  каналов/DM, открывается из нижней панели или по `Cmd+K`). У голосовых каналов
  показывается live-ростер участников.
- Управление голосом — в нижней панели **BottomBar**: кнопки микрофона, deafen,
  «live» (демонстрация), индикатор непрочитанных.
- Список участников голоса виден в правой панели **MembersRail** (раздел voice-
  участников по каналу).

### 2.2. Микрофон

Кнопка микрофона в BottomBar включает/выключает передачу голоса. Дополнительно:

- **Глобальный хоткей** `CommandOrControl+Shift+M` — переключает микрофон даже
  когда окно не в фокусе (реализовано через Electron-мост `window.chazh.onToggleMic`,
  см. [раздел 7](#7-интеграция-с-electron-главным-процессом)).
- Два режима работы микрофона (см. [раздел 4](#4-режимы-микрофона-voice-vs-ptt)):
  **голосовая активация** (`voice`, по умолчанию) и **Push-To-Talk** (`ptt`).

### 2.3. Deafen («заглушить всё»)

Кнопка deafen в BottomBar глушит **весь входящий звук** (голоса других участников
и саундпад). При включении deafen микрофон автоматически выключается; при выключении
deafen — предыдущее состояние микрофона восстанавливается. Другим участникам ваше
состояние транслируется через атрибут `deaf` (`'1'`/`'0'`) и отображается в их
списке участников.

### 2.4. Демонстрация экрана (Go Live)

- Кнопка «монитор» в BottomBar запускает демонстрацию. Выбор качества — в
  **VoiceSettingsModal** (`source` / `q1080` / `q720` / `q360`).
- На **Windows** можно захватывать системный звук (loopback). На **macOS** это
  невозможно — `getDisplayMedia` там не возвращает аудио-дорожку (см.
  [раздел 8](#8-демонстрация-экрана)).

### 2.5. Просмотр чужой демонстрации

Просмотр реализован в `src/features/main/ScreenSharePane.tsx`:

- Если демонстрируют несколько участников — сверху по центру появляются «пилюли»-
  кнопки для переключения активного экрана.
- Кнопка полноэкранного режима — справа сверху.
- Внизу слева — красная точка + подпись «X демонстрирует экран».

### 2.6. Саундпад

Общие звуковые клипы сервера, которые проигрываются в голосовой канал как
отдельная аудио-дорожка (см. [раздел 9](#9-саундпад-soundboard)). Есть отдельный
тумблер mute для саундпада, не зависящий от deafen.

### 2.7. Настройки голоса

Открываются в **VoiceSettingsModal**: выбор устройств ввода/вывода, привязка
PTT-клавиши, тумблеры шумоподавления / эхоподавления / AGC, качество демонстрации
экрана. Все настройки сохраняются в `localStorage` (см.
[раздел 10](#10-настройки-и-их-хранение)).

## 4. Режимы микрофона: voice vs PTT

Режим хранится в настройках (`mode: 'voice' | 'ptt'`).

### 4.1. Голосовая активация (`voice`, по умолчанию)

Микрофон всегда «открыт», но при заданном пороге включается voice-activation gate:
гейт активируется, если `micThreshold > 0`. При `micThreshold = 0` гейт отключён
(микрофон передаёт всё подряд).

### 4.2. Push-To-Talk (`ptt`)

Микрофон по умолчанию выключен; передача — пока удерживается PTT-клавиша.

- Клавиша задаётся в `pttKey` (по умолчанию `Space`).
- Поддерживаются и кнопки мыши: `pttKey` может быть `Mouse1`, `Mouse2`, `Mouse3`,
  `Mouse4` (индекс кнопки).

> **Gotcha (мышь + PTT):** для мыши код сверяет `e.button === parseInt(pttKey.slice(5))`
> — значение должно соответствовать нумерации `MouseEvent.button`.

> **Gotcha (гейт только в voice):** микрофонный гейт работает только в режиме
> `voice`, не в PTT. При переключении режима или отключении гейта дорожка
> переоткрывается (`mediaStreamTrack.enabled = true`), чтобы вернуть звук, если
> гейт ранее «придушил» дорожку.

## 5. Микрофонный гейт (порог голосовой активации)

Реализация в `src/lib/voice.ts`:

- Уровень звука измеряется через FFT (`fftSize = 512`), считается **RMS**.
- Шкала пользователя `0..1` нормализуется в `0..0.3` RMS (константа `MIC_RMS_FULL = 0.3`).
  Слайдер показывает нормализованную шкалу, индикатор уровня визуально совпадает со
  слайдером.
- **Hold-on time = 250 мс**: после окончания речи гейт держит дорожку открытой ещё
  250 мс, чтобы не обрезать «хвосты» слов.
- Для измерения используется **клон** микрофонной дорожки (публикуемая дорожка не
  затрагивается). Гейт управляет `mediaStreamTrack.enabled` на публикуемой дорожке —
  это дёшево и не требует ренеготиации (renegotiation) WebRTC.

## 6. Шумоподавление, эхоподавление и AGC

В клиенте есть две линии обработки звука: нейросетевой шумодав RNNoise (наш выбор по
умолчанию) и браузерные фильтры как fallback.

### 6.1. RNNoise (клиентский шумодав, тумблер «Шумоподавление»)

Реализация в `src/lib/rnnoise.ts`, библиотека `@sapphi-red/web-noise-suppressor`.

- WebAssembly-шумодав, работает **на клиенте** (никакого серверного лицензирования).
- Запускается на частоте **48 кГц** в отдельном `AudioContext` (не в контексте LiveKit).
- Интегрирован как LiveKit `TrackProcessor`
  (`TrackProcessor<Track.Kind.Audio, AudioProcessorOptions>`).
- Имя процессора: `RNNOISE_PROCESSOR_NAME = 'rnnoise'`.
- Использование: `track.setProcessor(createRnnoiseProcessor())` для включения и
  `track.stopProcessor()` для выключения.
- Инициализация: загрузка WASM → добавление AudioWorklet-модуля → создание
  `RnnoiseWorkletNode` → цепочка `source → node → destination`.
- Очистка: отключение нод, уничтожение ноды, закрытие `AudioContext`.

WASM/worklet-ресурсы (из пакета `@sapphi-red/web-noise-suppressor`):

| Ресурс | Что это |
|---|---|
| `rnnoiseWasmUrl` | WASM-бинарь шумодава |
| `rnnoiseSimdWasmUrl` | WASM-бинарь с SIMD |
| `rnnoiseWorkletUrl` | AudioWorklet-модуль |

**Тумблер в UI:** включается настройкой `noiseSuppression` (сохраняется в
`localStorage` ключ `chazh.voice`).

#### Почему RNNoise, а НЕ Krisp

Krisp (предыдущий вариант шумоподавления в LiveKit) **требует LiveKit Cloud** и на
self-hosted-инсталляции недоступен. Поскольку мы хостим LiveKit сами
(см. [раздел 1](#1-технологический-фундамент-livekit-self-hosted)), Krisp не
вариант. RNNoise — клиентский WASM-шумодав без серверной лицензии — заменяет Krisp
и работает полностью на стороне клиента.

> Зависимость `@livekit/krisp-noise-filter` всё ещё присутствует в `package.json`, но **не используется**
> (RNNoise её заменил). Удалять не обязательно — она может пригодиться, если проект переедет на LiveKit Cloud.

> **Gotcha (частота):** RNNoise требует `AudioContext` на 48 кГц и создаёт
> **отдельный** контекст (не контекст LiveKit). Если LiveKit использует другую
> частоту дискретизации — браузер ресемплит автоматически.

### 6.2. Fallback: браузерные echoCancellation / noiseSuppression

Если RNNoise недоступен, клиент откатывается на браузерные `echoCancellation` и
`noiseSuppression` из `AudioCaptureOptions`.

### 6.3. Эхоподавление и AGC

Отдельные тумблеры в настройках:

- `echoCancellation` — эхоподавление (по умолчанию `true`).
- `autoGain` — автоматическая регулировка усиления, AGC (по умолчанию `true`).

## 7. Интеграция с Electron (главным процессом)

Голос частично завязан на нативный мост `window.chazh` (preload-бридж; существует
только в Electron, в браузере его нет):

- `window.chazh.setMicHotkey(accel)` / `window.chazh.onToggleMic(cb)` — глобальный
  хоткей переключения микрофона `CommandOrControl+Shift+M`, работающий вне фокуса окна.
- `window.chazh.setShareAudio(on)` — включение loopback системного звука на Windows
  перед `getDisplayMedia` (см. [раздел 8](#8-демонстрация-экрана)).

> **Gotcha:** `window.chazh` есть только в Electron. В обычном браузере нативные
> фичи (глобальный хоткей, системный звук) недоступны — проверяйте существование
> `window.chazh` перед вызовом (`window.chazh?.…`).

## 8. Демонстрация экрана

### 8.1. Запуск (Go Live) и качество

Пресеты качества (хранятся в `localStorage`, ключ настройки `screenQuality`):

| Пресет | Разрешение / FPS | Комментарий |
|---|---|---|
| `source` | нативное разрешение, максимальный битрейт | `ScreenSharePresets.original` (без ограничений) |
| `q1080` | 1080p @ 30 fps | |
| `q720` | 720p @ 30 fps | значение по умолчанию (`DEFAULTS`) |
| `q360` | 360p @ 15 fps | экономичный режим |

### 8.2. Системный звук (loopback) — только Windows

- Флаг `screenAudio`. При его включении на Windows системный звук подмешивается в
  захват (loopback).
- На **macOS** `getDisplayMedia` **никогда** не возвращает аудио-дорожку — системный
  звук недоступен.
- Перед `getDisplayMedia` клиент вызывает `window.chazh?.setShareAudio(bool)`, чтобы
  главный процесс настроил loopback. Код проверяет `window.chazh?.platform === 'win32'`
  прежде чем запрашивать аудио в constraints.

> **Gotcha (порядок вызовов):** обработчик `setDisplayMediaRequestHandler` должен быть
> установлен **до** `getDisplayMedia`. Если меняются настройки демонстрации (качество
> или звук) на лету — нужно остановить и перезапустить демонстрацию
> (`restartScreen()`), новые настройки применяются без повторного выбора источника
> пользователем.

### 8.3. Просмотр чужой демонстрации

`src/features/main/ScreenSharePane.tsx`:

- Дорожка: `Track.Kind.Video` с источником `Track.Source.ScreenShare`.
- `<video>` с `autoPlay`, `objectFit: contain`; attach/detach по жизненному циклу дорожки.
- Мультиэкран: переключение «пилюлями» вверху по центру.
- Полноэкранный режим: кнопка справа сверху.
- Live-индикатор: красная точка + «X демонстрирует экран» внизу слева.

## 9. Саундпад (Soundboard)

Реализация в `src/lib/soundboard.ts`. Это общие для сервера звуковые клипы, которые
играются прямо в голосовой канал.

- Публикуется **отдельной** аудио-дорожкой с именем `'soundboard'` (по этому имени
  принимающая сторона отличает её от обычной голосовой дорожки).
- Источник: `soundboard.outputTrack()` возвращает `MediaStreamTrack` (клон публикуется
  в канал).
- Mute: отдельный тумблер `soundboardMuted` — глушит саундпад независимо от deafen.
- Входящие саундпад-дорожки хранятся отдельно (в карте `soundboardEls`).

Управление клипами (fetch/upload) — через бэкенд (presigned S3 PUT); подробности
эндпоинтов см. в [docs/BACKEND-CONTRACT.md](./BACKEND-CONTRACT.md).

> **Gotcha:** саундпад публикуется как **клон** (не оригинал), поэтому `stop()` от
> LiveKit при отключении не затрагивает постоянный микшер приложения. Принимающая
> сторона идентифицирует дорожку по `trackName = 'soundboard'`.

## 10. Настройки и их хранение

Все настройки голоса хранятся в `localStorage` под ключом **`chazh.voice'`**
(дополнительно громкости отдельных участников — под ключом `chazh.voice.vol`).
Серверной синхронизации профиля нет.

### 10.1. Схема `VoiceSettings`

| Поле | Тип | Назначение | По умолчанию |
|---|---|---|---|
| `inputId` | string | ID устройства ввода (пусто = default) | `''` |
| `outputId` | string | ID устройства вывода | `''` |
| `noiseSuppression` | boolean | RNNoise-шумодав («Шумоподавление») | `true` |
| `echoCancellation` | boolean | эхоподавление | `true` |
| `autoGain` | boolean | AGC (авторегулировка усиления) | `true` |
| `mode` | `'voice' \| 'ptt'` | режим микрофона | `'voice'` |
| `pttKey` | string | PTT-клавиша/кнопка мыши | `'Space'` |
| `micThreshold` | number `0..1` | порог гейта (`0` = выкл.) | `0` |
| `soundboardMuted` | boolean | mute саундпада | `false` |
| `screenQuality` | `'source' \| 'q1080' \| 'q720' \| 'q360'` | качество демонстрации | `'q720'` |
| `screenAudio` | boolean | системный звук при демонстрации (Windows) | `false` |

Заводские значения определены в объекте `DEFAULTS` в `voice.ts`.

> **Gotcha:** настройки живут только в `localStorage`. При очистке localStorage или
> смене браузера/устройства они теряются.

### 10.2. Громкость отдельных участников

- Индивидуальный множитель громкости на пользователя: `0..2`.
- Хранится в `localStorage` ключ `chazh.voice.vol` как `identity → multiplier`.

## 11. Выбор устройств

- **Ввод:** `inputId` (пустая строка = устройство по умолчанию). Применяется через
  `room.switchActiveDevice('audioinput', id)` + `captureOpts.deviceId`.
- **Вывод:** `outputId`. Применяется через `room.switchActiveDevice('audiooutput', id)`
  + `el.setSinkId(id)` на всех аудио-элементах.
- **Разблокировка меток устройств:** `requestMicPermission()` запрашивает
  `getUserMedia`, чтобы получить разрешение и видеть человекочитаемые `label` в
  `enumerateDevices`.
- **Перечисление:** `listDevices()` → `{ inputs: AudioDevice[], outputs: AudioDevice[] }`
  (каждое — `{ deviceId, label }`).

> **Gotcha (метки пустые):** `label` остаётся пустым, пока пользователь не выдал
> разрешение на микрофон. Вызывайте `requestMicPermission()` до `listDevices()`,
> иначе метки будут пустыми.

> **Gotcha (`setSinkId`):** переключение устройства вывода через `setSinkId()`
> работает только если элемент находится в документе (не скрыт). Поэтому приложение
> добавляет аудио-элементы в `document.body`.

## 12. Состояние участников голоса

Тип `VoiceParticipant`:

```ts
{ id, name, speaking, micOn, deafened, volume }
```

- **Определение говорящего (`speaking`):** список активных спикеров приходит событием
  LiveKit; в UI отмечается визуальным индикатором.
- **`deafened`:** читается из атрибутов участника (`deaf: '1'`), отображается в списке.
- **`volume`:** индивидуальный множитель `0..2` (см. [раздел 10.2](#102-громкость-отдельных-участников)).

## 13. Чек-лист «частых граблей»

- `window.chazh` есть только в Electron — проверяйте `window.chazh?.…`.
- Системный звук демонстрации — **только Windows**; на macOS аудио из `getDisplayMedia`
  не приходит вообще.
- Метки устройств пустые до выдачи разрешения на микрофон — сначала
  `requestMicPermission()`, потом `listDevices()`.
- `setSinkId()` для устройства вывода требует, чтобы элемент был в DOM.
- Микрофонный гейт работает только в режиме `voice`; при смене режима дорожка
  переоткрывается.
- RNNoise — единственный наш шумодав (Krisp требует LiveKit Cloud, недоступен на
  self-host).
- Настройки голоса хранятся только локально (`localStorage` `chazh.voice`),
  серверной синхронизации нет.
- Смена качества/звука демонстрации на лету = стоп + рестарт демонстрации.

> Строка хоткея для `setMicHotkey` — это стандартный [Electron Accelerator](https://www.electronjs.org/docs/latest/api/accelerator)
> (например `CommandOrControl+Shift+M`, дефолт): модификаторы `CommandOrControl`/`Alt`/`Shift`/`Super` + клавиша,
> через `+`. Регистрируется в `electron/main.ts` через `globalShortcut.register` (глобально, вне фокуса окна).
